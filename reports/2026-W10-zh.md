# 周报 2026-W10 | LULESH 2.0 现代化重构与 Kokkos 性能优化

**周期**：2026-03-02 ~ 2026-03-08
**项目**：LULESH 2.0 HPC 代理应用现代化改造
**平台**：macOS Apple Silicon / Kokkos 5.0.2 OpenMP 后端
**基准**：`OMP_PROC_BIND=close ./lulesh2.0 -s 10 -i 50` → `Final Origin Energy = 8.104796e+04`

---

## 本周工作概览

本周完成了 LULESH 2.0 从原始单文件代码库到现代 C++20 + Kokkos 5 并行框架的全流程改造，
并通过两轮内核融合将性能提升至原版的 **2.15×**，最后完成项目工程化整理。

---

## 一、模块化重构

**起点**：原版 `lulesh.cc` 单文件约 2813 行，MPI + OpenMP pragma 混合并行，无法独立分析各物理模块。

**拆分策略**：按物理职责将代码切分为 10 个独立模块（+1 个 main 入口），每个模块一对 `.cc/.h`：

| 模块 | 职责 |
|------|------|
| `lulesh-geometry` | 形函数、体积/面积/特征长度 |
| `lulesh-stress` | 应力积分、Flanagan-Belytschko 沙漏控制 |
| `lulesh-nodal` | 力→加速度→速度→位置 |
| `lulesh-kinematics` | 速度梯度、变形率张量 |
| `lulesh-viscosity` | 单调人工黏性 Q |
| `lulesh-eos` | 物态方程、材料属性更新 |
| `lulesh-timestep` | Courant/水力学时间步约束 |
| `lulesh-integration` | Leap-Frog 主循环 |
| `lulesh-init` | 网格初始化、Domain 构造 |
| `lulesh-util` | 命令行解析、结果输出 |

同步移除全部 MPI 代码与 Silo/VisIt 可视化依赖，构建系统从 Makefile 迁移到 **CMake 3.16+**。

---

## 二、现代 C++20 改造

在正确性基准保持不变的前提下，逐项应用 C++20 特性：

| 改动 | 旧写法 | 新写法 |
|------|--------|--------|
| 空指针 | `NULL` | `nullptr` |
| 类型别名 | `typedef Real_t double` | `using Real_t = double` |
| C 标准头文件 | `<math.h>` | `<cmath>` |
| BC 标志位常量 | `#define FREEX 0x1` | `constexpr Int_t FREEX = 0x1` |
| 取最大值 | `MAX(a,b)` 宏 | `std::max(a,b)` |
| 数学函数封装 | `SQRT(x)` 内联包装 | 直接调用 `std::sqrt(x)` |
| 内存管理 | `Allocate<T>` / `Release<T>` 裸指针模板 | `std::vector<T>` |

**Domain 类重组**：将约 80 个私有成员按语义归并为 3 个嵌套结构体：
`NodeArrays`（节点数组）、`ElemConnectivity`（连接关系）、`ElemState`（单元状态）。

## 三、Kokkos 5 迁移

**目标**：以 Kokkos 5（OpenMP 后端）替换全部 `#pragma omp parallel for`，
为后续 GPU 移植保留统一接口。

迁移分三批完成：

**Batch A — 基础设施**
- `CMakeLists.txt` 引入 `find_package(Kokkos 5.0 REQUIRED)`
- `Domain` 的 `std::vector<T>` 成员字段全部替换为 `Kokkos::View<T*>`（`m_regElemlist` 保留嵌套 vector）
- `main()` 用 `Kokkos::initialize` / `Kokkos::finalize` 包裹

**Batch B — 并行循环**
- `#pragma omp parallel for` → `Kokkos::parallel_for("label", N, [&](Index_t i){...})`
- 关键选择：使用 `[&]` 捕获而非 `KOKKOS_LAMBDA`（后者等价于 `[=]`，在 CPU 同步场景下会引入不必要的 const 限制）
- 多段 `omp for nowait` 拆分为独立 `Kokkos::parallel_for`
- `std::vector` 临时数组：在 lambda 外提取 `.data()` 指针后传入（C++ 捕获语义要求）

**Batch C — 归约操作**
- 手动线程局部 min 数组 → `Kokkos::MinLoc<Real_t, Index_t>` reducer
- 后期优化为 `Kokkos::Min<Real_t>` 双 reducer（因原始代码从不使用 `.loc` 字段）

## 四、性能优化

通过 macOS `sample` 工具采样分析，确认 **OMP barrier 开销占运行时间约 79.5%**，
共实施两轮 9 项内核融合优化，总计减少约 27 个 barriers/step。

### 第一轮（Opt-1~3）：大体量融合

| 优化 | 文件 | 核心改动 | 节省 |
|------|------|---------|------|
| Opt-1 沙漏融合 | `lulesh-stress.cc` | 删除 `CalcFBHourglassForceForElems`，散步逻辑内联进 `CalcHourglassControlForElems` | 1 barrier + ~35 MB 临时向量（s=45） |
| Opt-2 EvalEOS 准备阶段 | `lulesh-eos.cc` | 5 个 gather/compression/clamp/work 内核 → 1 个 | 4 barriers/rep/region |
| Opt-3 CalcEnergy 超级融合 | `lulesh-eos.cc` | 11 个子内核（e1/q1/e2/e3/q2 + 3×CalcPressure×2）→ 1 个，删除 `CalcPressureForElems` | 10 barriers/rep + 1 向量 |

**第一轮结果**：7.10 s → 3.65 s，**加速比 2.14×**

### 第二轮（Opt-4~9）：细粒度融合

| 优化 | 文件 | 核心改动 | 节省 |
|------|------|---------|------|
| Opt-4 速度+位置融合 | `lulesh-nodal.cc` | `CalcVelocityForNodes` + `CalcPositionForNodes` → 1 个内核 | 1 barrier |
| Opt-5 双 reduce 融合 | `lulesh-timestep.cc` | Courant + Hydro 两个独立 MinLoc reduce → 1 个 `Kokkos::Min<Real_t>` 变参 reduce | 11 barriers/step |
| Opt-6 scatter+声速融合 | `lulesh-eos.cc` | `EvalEOSForElems_scatter` + `CalcSoundSpeedForElems` → 1 个内核，删除后者函数 | 11 barriers/step |
| Opt-7 材料属性融合 | `lulesh-eos.cc` | eosvmin clamp + eosvmax clamp + 体积检查 3 个内核 → 1 个 | 2 barriers |
| Opt-8 InitStress 内联 | `lulesh-stress.cc` | 删除 `InitStressTermsForElems`，`sig = -p-q` 直接在 scatter 内核中计算 | 1 barrier + 3×numElem 临时向量 |
| Opt-9 运动学+Lagrange 融合 | `lulesh-kinematics.cc` | `CalcKinematicsForElems` + `CalcLagrangeElements` → 1 个内核，D[0/1/2] 保留为栈局部变量 | 1 barrier |

**第二轮结果**：3.65 s → **3.30 s**，累计加速比 **2.15×**

### 综合性能数据（`-s 45 -i 200`，RelWithDebInfo -O2）

| 阶段 | 耗时 | FOM（z/s） | 加速比 |
|------|------|-----------|--------|
| 原始 LULESH 2.0 | 7.10 s | 2572 | 1.00× |
| 第一轮优化后 | 3.65 s | 5498 | 2.14× |
| **第二轮优化后** | **3.30 s** | **5518** | **2.15×** |

## 关键决策记录

1. **`[&]` vs `KOKKOS_LAMBDA`**：CPU-only 同步 Kokkos 使用 `[&]` 引用捕获，避免 `[=]` 带来的 const 闭包限制，同时无需 GPU 兼容。

2. **Opt-5 改用 `Kokkos::Min` 而非 `MinLoc`**：原始 Courant/Hydro reduce 从不使用结果的 `.loc` 字段，改用 `Kokkos::Min<Real_t>` 变参 reducer，单次 reduce 同时得到两个最小值，减少 1 个 barrier/region。

3. **Opt-8 正确性保证**：`CalcSoundSpeedForElems` 读取的是临时指针 `e_new_ptr`/`p_new_ptr` 而非 `domain.e/p`，因此与 scatter 写操作不存在数据竞争，可安全融合。

4. **`AllocateGradients`/`AllocateStrains` 保留为 public**：两者在 `lulesh-viscosity.cc` 和 `lulesh-kinematics.cc` 中从 Domain 外部调用，仅将只在构造函数内部使用的 `AllocateNodePersistent`/`AllocateElemPersistent` 设为 private。

## 遇到的问题

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 编译报错：`AllocateGradients` is private | 误将外部调用的 Allocate* 方法移至 private | 将从外部 .cc 文件调用的 4 个方法保留为 public |
| Kokkos::View `.empty()` 不存在 | Kokkos::View 无 STL 兼容接口 | 改用 `.extent(0) == 0` 判断空 View |
| `std::vector` 临时指针在 lambda 内失效 | C++ 不允许 lambda 捕获 vector 本身后再用 `.data()` | 在 lambda 外先 `.data()` 取指针，lambda 捕获指针 |

## 下周计划

- 调研在 Kokkos 框架下替代 MPI 的分布式并行方案，候选方向包括：
  - **Kokkos Remote Spaces**（`kokkos-remote-spaces`）：通过 PGAS 模型为 Kokkos::View 提供跨进程的全局地址空间访问，无需显式 MPI 调用
  - **Kokkos + MPI 混合**：保留 MPI 负责跨节点通信，Kokkos 负责节点内并行，评估通信边界的抽象成本
  - **HPX / SHMEM 后端**：考察 Kokkos 是否可直接绑定其他分布式运行时以替代 MPI 语义
