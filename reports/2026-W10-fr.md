# Rapport hebdomadaire 2026-W10 | LULESH 2.0 : refactorisation moderne et optimisation des performances avec Kokkos

**Période** : 2026-03-02 ~ 2026-03-08
**Projet** : Modernisation de l'application proxy HPC LULESH 2.0
**Plateforme** : macOS Apple Silicon / backend OpenMP Kokkos 5.0.2
**Référence de correction** : `OMP_PROC_BIND=close ./lulesh2.0 -s 10 -i 50` → `Final Origin Energy = 8.104796e+04`

---

## Vue d'ensemble de la semaine

Cette semaine, la migration complète de LULESH 2.0 — depuis une base de code monolithique jusqu'à un framework parallèle C++20 + Kokkos 5 — a été achevée. Deux rondes de fusion de noyaux ont permis d'atteindre un gain de performance de **2,15×** par rapport à la version originale. Le projet a ensuite fait l'objet d'une réorganisation technique finale.

---

## I. Refactorisation modulaire

**Point de départ** : le fichier unique `lulesh.cc` (~2 813 lignes), combinant MPI et pragma OpenMP, rendait toute analyse indépendante des modules physiques impossible.

**Stratégie de découpage** : le code a été réparti en 10 modules indépendants (+1 point d'entrée `main`), chacun constitué d'une paire `.cc/.h` :

| Module | Responsabilité |
|--------|----------------|
| `lulesh-geometry` | Fonctions de forme, volume / aire / longueur caractéristique |
| `lulesh-stress` | Intégration des contraintes, contrôle sablier Flanagan-Belytschko |
| `lulesh-nodal` | Force → accélération → vitesse → position |
| `lulesh-kinematics` | Gradient de vitesse, tenseur de taux de déformation |
| `lulesh-viscosity` | Viscosité artificielle Q monotone |
| `lulesh-eos` | Équation d'état, mise à jour des propriétés matérielles |
| `lulesh-timestep` | Contraintes de pas de temps Courant / hydrodynamique |
| `lulesh-integration` | Boucle principale Leap-Frog |
| `lulesh-init` | Initialisation du maillage, construction du Domain |
| `lulesh-util` | Analyse de la ligne de commande, sortie des résultats |

L'intégralité du code MPI et des dépendances de visualisation Silo/VisIt a été supprimée. Le système de build a été migré de Makefile vers **CMake 3.16+**.

---

## II. Modernisation C++20

Les fonctionnalités C++20 ont été appliquées point par point, sans altérer la référence de correction :

| Modification | Ancienne écriture | Nouvelle écriture |
|--------------|-------------------|-------------------|
| Pointeur nul | `NULL` | `nullptr` |
| Alias de type | `typedef Real_t double` | `using Real_t = double` |
| En-têtes C standard | `<math.h>` | `<cmath>` |
| Constantes de drapeaux BC | `#define FREEX 0x1` | `constexpr Int_t FREEX = 0x1` |
| Maximum | macro `MAX(a,b)` | `std::max(a,b)` |
| Encapsuleurs de fonctions math. | wrapper inline `SQRT(x)` | appel direct `std::sqrt(x)` |
| Gestion mémoire | templates `Allocate<T>` / `Release<T>` sur pointeurs bruts | `std::vector<T>` |

**Réorganisation de la classe `Domain`** : environ 80 membres privés regroupés sémantiquement en 3 structures imbriquées :
`NodeArrays` (tableaux nodaux), `ElemConnectivity` (relations de connectivité), `ElemState` (état des éléments).

## III. Migration vers Kokkos 5

**Objectif** : remplacer tous les `#pragma omp parallel for` par Kokkos 5 (backend OpenMP), en conservant une interface unifiée pour une future portabilité GPU.

La migration a été réalisée en trois lots :

**Lot A — Infrastructure**
- `CMakeLists.txt` : ajout de `find_package(Kokkos 5.0 REQUIRED)`
- Les champs `std::vector<T>` du `Domain` sont entièrement remplacés par `Kokkos::View<T*>` (`m_regElemlist` conserve un vector imbriqué)
- `main()` encapsulé dans `Kokkos::initialize` / `Kokkos::finalize`

**Lot B — Boucles parallèles**
- `#pragma omp parallel for` → `Kokkos::parallel_for("label", N, [&](Index_t i){...})`
- Choix clé : capture `[&]` préférée à `KOKKOS_LAMBDA` (équivalent à `[=]`, qui introduit une restriction `const` inutile dans un contexte synchrone CPU)
- Les segments `omp for nowait` multiples sont séparés en `Kokkos::parallel_for` indépendants
- Tableaux temporaires `std::vector` : le pointeur `.data()` est extrait avant le lambda (exigence de la sémantique de capture C++)

**Lot C — Réductions**
- Tableaux min locaux par thread → reducer `Kokkos::MinLoc<Real_t, Index_t>`
- Optimisé ultérieurement en double reducer `Kokkos::Min<Real_t>` (le champ `.loc` du résultat n'est jamais utilisé dans le code d'origine)

## IV. Optimisations des performances

Une analyse par échantillonnage avec l'outil `sample` de macOS a confirmé que **la surcharge des barrières OMP représentait environ 79,5 % du temps d'exécution**. Deux rondes d'optimisation, totalisant 9 fusions de noyaux, ont permis de réduire d'environ 27 barrières par pas de temps.

### Première ronde (Opt-1~3) : fusions massives

| Optimisation | Fichier | Modification principale | Économie |
|--------------|---------|------------------------|----------|
| Opt-1 : fusion sablier | `lulesh-stress.cc` | Suppression de `CalcFBHourglassForceForElems`, logique de scatter intégrée dans `CalcHourglassControlForElems` | 1 barrière + ~35 Mo de vecteurs temporaires (s=45) |
| Opt-2 : phase de préparation EvalEOS | `lulesh-eos.cc` | 5 noyaux gather/compression/clamp/work → 1 seul | 4 barrières/rep/région |
| Opt-3 : super-fusion CalcEnergy | `lulesh-eos.cc` | 11 sous-noyaux (e1/q1/e2/e3/q2 + 3×CalcPressure×2) → 1 seul, suppression de `CalcPressureForElems` | 10 barrières/rep + 1 vecteur |

**Résultat de la première ronde** : 7,10 s → 3,65 s, **accélération 2,14×**

### Deuxième ronde (Opt-4~9) : fusions à grain fin

| Optimisation | Fichier | Modification principale | Économie |
|--------------|---------|------------------------|----------|
| Opt-4 : fusion vitesse + position | `lulesh-nodal.cc` | `CalcVelocityForNodes` + `CalcPositionForNodes` → 1 noyau | 1 barrière |
| Opt-5 : fusion double reduce | `lulesh-timestep.cc` | 2 reduces MinLoc Courant + Hydro → 1 reduce variadique `Kokkos::Min<Real_t>` | 11 barrières/step |
| Opt-6 : fusion scatter + vitesse du son | `lulesh-eos.cc` | `EvalEOSForElems_scatter` + `CalcSoundSpeedForElems` → 1 noyau, suppression de la seconde fonction | 11 barrières/step |
| Opt-7 : fusion propriétés matérielles | `lulesh-eos.cc` | clamp eosvmin + clamp eosvmax + vérification volume (3 noyaux) → 1 seul | 2 barrières |
| Opt-8 : inlining InitStress | `lulesh-stress.cc` | Suppression de `InitStressTermsForElems`, `sig = -p-q` calculé directement dans le noyau scatter | 1 barrière + 3×numElem vecteurs temporaires |
| Opt-9 : fusion cinématique + Lagrange | `lulesh-kinematics.cc` | `CalcKinematicsForElems` + `CalcLagrangeElements` → 1 noyau, D[0/1/2] conservés en variables locales de pile | 1 barrière |

**Résultat de la deuxième ronde** : 3,65 s → **3,30 s**, accélération cumulée **2,15×**

### Données de performance consolidées (`-s 45 -i 200`, RelWithDebInfo -O2)

| Étape | Durée | FOM (z/s) | Accélération |
|-------|-------|-----------|--------------|
| LULESH 2.0 original | 7,10 s | 2572 | 1,00× |
| Après première ronde | 3,65 s | 5498 | 2,14× |
| **Après deuxième ronde** | **3,30 s** | **5518** | **2,15×** |

## V. Réorganisation du projet

**Nettoyage du code**
- Suppression de 8 directives `#include` inutiles dans `apps/lulesh_main.cc` (`<climits>`, `<iostream>`, etc.)
- Suppression d'un bloc de 3 lignes de commentaires de débogage résiduels
- Remplacement de tous les gardes d'inclusion `#ifndef` par `#pragma once`
- Suppression des macros obsolètes `CACHE_COHERENCE_PAD_REAL` / `CACHE_ALIGN_REAL` et des membres associés
- Migration de `struct cmdLineOpts` de `lulesh.h` vers `lulesh-util.h` (respect du principe de responsabilité unique)

**Réorganisation de l'arborescence**
```
apps/lulesh_main.cc     ← main() extrait de src/
scripts/build.sh        ← script de build CMake en une commande (avec paramètres Kokkos/OpenMP macOS)
scripts/run.sh          ← deux modes d'exécution : verify / bench
samples/                ← archivage des sorties d'analyse par échantillonnage
tests/baselines/        ← valeurs de référence pour la vérification de correction
```

**Réécriture du README** : la version originale a été remplacée par une documentation complète incluant les dépendances, les commandes de build détaillées, les tableaux de performances et le résumé des deux rondes d'optimisation.

## Décisions techniques clés

1. **`[&]` vs `KOKKOS_LAMBDA`** : en contexte Kokkos synchrone CPU-only, la capture par référence `[&]` est préférée pour éviter la restriction `const` introduite par `[=]`, sans nécessiter de compatibilité GPU.

2. **Opt-5 : `Kokkos::Min` plutôt que `MinLoc`** : les reduces Courant et Hydro n'utilisent jamais le champ `.loc` du résultat. L'usage du reducer variadique `Kokkos::Min<Real_t>` permet d'obtenir deux minima en un seul appel, économisant 1 barrière par région.

3. **Garantie de correction pour Opt-8** : `CalcSoundSpeedForElems` lit les pointeurs temporaires `e_new_ptr` / `p_new_ptr` et non `domain.e/p` ; il n'existe donc aucune course aux données avec les écritures scatter, rendant la fusion sûre.

4. **`AllocateGradients` / `AllocateStrains` conservées en public** : ces deux méthodes sont appelées depuis des fichiers `.cc` externes (`lulesh-viscosity.cc`, `lulesh-kinematics.cc`) ; seules `AllocateNodePersistent` et `AllocateElemPersistent`, utilisées exclusivement dans le constructeur, sont passées en `private`.

## Problèmes rencontrés

| Problème | Cause | Solution |
|----------|-------|---------|
| Erreur de compilation : `AllocateGradients` is private | Déplacement erroné de méthodes appelées en externe vers `private` | Les 4 méthodes appelées depuis d'autres `.cc` sont maintenues en `public` |
| `Kokkos::View` ne possède pas de méthode `.empty()` | `Kokkos::View` n'implémente pas l'interface STL | Remplacement par `.extent(0) == 0` |
| Pointeur temporaire `std::vector` invalide dans le lambda | Un lambda ne peut pas capturer un `vector` puis appeler `.data()` à l'intérieur | Extraction du pointeur via `.data()` avant le lambda ; le lambda capture le pointeur brut |

## Objectifs de la semaine prochaine

- Étudier les solutions de parallélisme distribué en remplacement de MPI dans le cadre Kokkos, parmi lesquelles :
  - **Kokkos Remote Spaces** (`kokkos-remote-spaces`) : accès à un espace d'adressage global inter-processus pour `Kokkos::View` via le modèle PGAS, sans appels MPI explicites
  - **Hybride Kokkos + MPI** : MPI conservé pour les communications inter-nœuds, Kokkos pour le parallélisme intra-nœud ; évaluation du coût d'abstraction aux frontières de communication
  - **Backend HPX / SHMEM** : faisabilité d'un binding direct de Kokkos à d'autres runtimes distribués en substitution de la sémantique MPI
